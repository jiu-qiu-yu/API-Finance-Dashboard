## Context

当前问题发生在使用本地浏览器用户数据目录复用登录态的抓取链路上：点击“测试”后，页面确实进入了用户账号，但 Chrome 随即显示“您使用的是不受支持的命令行标记：--no-sandbox”，页面停在告警页或启动页，后续抓取逻辑无法继续。该问题同时影响单站点测试抓取和完整巡检，因为两条链路最终都依赖 `BrowserEngine` 启动持久化浏览器上下文。

约束如下：
- 必须继续复用用户真实 Chrome/Edge 配置目录，不能退回无状态登录。
- 不能再依赖 Playwright 静默回退到内置 Chromium，否则会触发用户配置降级问题。
- 必须优先避免会导致 Chrome 告警或冻结的启动参数，尤其是对真实用户 profile 的持久化启动。
- UI 需要给出可操作的失败原因，而不是只暴露 Playwright 原始异常。

## Goals / Non-Goals

**Goals:**
- 统一测试抓取和巡检流程的浏览器启动参数来源。
- 禁止对真实用户 profile 使用会触发不支持警告或稳定性问题的启动参数。
- 启动前尽早识别“浏览器可执行文件缺失 / profile 冲突 / 参数不兼容”等问题，并转成可理解的错误。
- 保持现有登录态复用能力，不要求用户重新输入账号密码。

**Non-Goals:**
- 不引入新的自动登录机制。
- 不修改抓取解析规则本身（CSS/正则提取逻辑不在本次范围内）。
- 不支持 Linux 容器场景下必须依赖 `--no-sandbox` 的运行模式；本次面向本地桌面应用真实浏览器启动。

## Decisions

### 1. 将浏览器启动策略集中到 `BrowserEngine`
- 决定：由 `BrowserEngine` 作为唯一浏览器启动入口，统一生成 `launch_persistent_context` 的 `executable_path`、`user_data_dir`、`profile-directory` 与自定义参数；测试抓取与巡检只传递配置，不各自拼装参数。
- 原因：当前问题跨越 `settings_panel.py`、`inspection_worker.py`、`inspection_service.py`，如果每条链路各自维护参数，容易再次出现一边修复一边遗漏。
- 备选方案：在 UI 层分别修复测试按钮和巡检按钮的参数。放弃原因是重复逻辑太多，后续容易漂移。

### 2. 对真实本地浏览器显式禁用 `--no-sandbox`
- 决定：在 Windows 本地桌面场景中，若使用真实 Chrome/Edge 可执行文件与用户 profile，启动参数中不得传入 `--no-sandbox`；如 Playwright 或调用侧存在该参数，需在进入浏览器前过滤掉。
- 原因：用户当前冻结现象直接与该参数相关，且 Chrome 已明确显示其为不受支持标记；继续携带该参数会损害稳定性。
- 备选方案：保留 `--no-sandbox` 并通过页面脚本绕过告警。放弃原因是治标不治本，而且会继续处于不稳定状态。

### 3. 保持“必须使用真实浏览器可执行文件”的硬约束
- 决定：启动前解析并校验系统已安装的 Chrome/Edge 可执行文件；若未找到则直接失败，不允许回退到 Playwright bundled Chromium。
- 原因：上一次故障已证明 bundled Chromium 会与用户 profile 版本不匹配，导致 downgrade 相关错误甚至配置损坏。
- 备选方案：找不到本地浏览器时自动回退。放弃原因是风险远高于收益。

### 4. 新增启动诊断层，区分“参数告警”和“页面业务卡死”
- 决定：在浏览器启动与首个页面加载阶段增加诊断：优先识别不支持参数告警页、profile 冲突、浏览器进程提前退出等启动级错误，并返回中文原因；只有成功通过启动诊断后，才进入页面抓取等待。
- 原因：当前用户能看到登录页面但不知道为什么“卡住”，需要把浏览器启动问题和站点自身慢加载区分开。
- 备选方案：继续仅依赖超时异常。放弃原因是无法指导用户处理。

### 5. 让测试抓取和巡检共用同一份启动配置传递链
- 决定：配置仓库中的浏览器可执行路径、profile 路径、profile 目录由两条流程完全复用，避免“测试可用、巡检异常”或反之。
- 原因：当前相关改动已涉及全链路传参，本次设计明确把它作为规范要求固定下来。
- 备选方案：测试抓取动态探测，巡检使用持久配置。放弃原因是行为不一致，难排障。

## Risks / Trade-offs

- [某些环境历史上依赖 `--no-sandbox`] → 明确将本次支持范围限定为本地桌面真实浏览器；如未来需要容器支持，再设计单独模式。
- [更严格地禁止回退会让“未安装 Chrome/Edge”的用户直接失败] → 提供明确错误提示和“自动检测浏览器”引导，优先保证数据安全与可诊断性。
- [新增诊断可能让启动流程多一层状态判断] → 诊断仅覆盖启动早期关键节点，避免引入复杂状态机。
- [真实页面卡慢仍可能被用户感知为冻结] → 在启动诊断通过后保留现有抓取超时与错误提示，并区分“浏览器已启动 / 页面仍在加载”。

## Migration Plan

- 修改 `BrowserEngine` 启动参数构建逻辑，确保过滤 `--no-sandbox` 并保留必要参数抑制扩展冲突。
- 统一测试抓取与巡检流程的浏览器配置传递。
- 增加启动诊断与中文错误映射。
- 通过现有测试补充启动配置相关断言；手动验证“测试抓取”和“开始巡检”两条路径。
- 若修复引入新的启动失败，可回滚到上一个稳定版本的启动参数构造，但必须保留“禁止 bundled Chromium 回退”的保护。

## Open Questions

- 当前 `--no-sandbox` 是由项目代码显式传入，还是被外部配置/历史缓存注入？实现前需要在代码中定位其真实来源。
- Chrome 告警页出现后，是否还能通过 DOM/标题稳定识别，以便给出更明确的诊断信息？若不能，则至少基于启动参数和进程退出信号判断。
- 是否需要在 UI 上额外展示“已使用的浏览器可执行文件路径”，帮助用户确认不是 Playwright bundled Chromium？
